RECO SDK¶

RECO SDK is based on CoreLocation.framework provided by iOS and conforms to the iBeacon standard. Region monitoring and region ranging are the two basic functions provided by this SDK. A beacon region is an area defined by the device’s proximity to Bluetooth low-energy beacons. The only role of a RECOBeacon is to broadcast the bluetooth signal containing data which is complying with the iBeacon standard. In iOS, regions associated with your app are tracked at all times, including when the app isn’t running. If a region boundary is crossed while an app isn’t running, that app is relaunched into the background to handle the event.

Monitoring RECOBeaconRegion¶

Beacon region monitoring uses an iOS device’s onboard radio to detect when the user is in the vicinity of Bluetooth low-energy devices that are advertising iBeacon information. You can use this capability to generate alerts or to provide other relevant information when the user enters or exits a beacon region. A beacon region is identified by the device’s proximity to Bluetooth low-energy beacons that advertise a combination of the following values:

  • Proximity UUID (universally unique identifier) which is a 128-bit value that uniquely identifies one or more beacons as a certain type or from a certain organization
  • Major : 16-bit unsigned integer that can be used to group related beacons that have the same proximity UUID
  • Minor : 16-bit unsigned integer that differentiates beacons with the same proximity UUID and major value

Because a single beacon region can represent multiple beacons, beacon region monitoring supports several interesting use cases. For example, an app dedicated to enhancing the experience of customers at a particular department store can use the same proximity UUID to monitor all stores in the department store chain. When the user approaches a store, the app detects the store’s beacons and uses the major and minor values of those beacons to determine additional information, such as which specific store was encountered or which section of the store the user is in. (Note that although every beacon must advertise a proximity UUID, major and minor values are optional.)

Defining a Beacon Region to Be Monitored¶

To begin monitoring a beacon region, define the region and register it with the system. You define a beacon region with the appropriate initialization method of the RECOBeaconRegion class. When you create a RECOBeaconRegion object, you specify the proximityUUID, major, and minor properties of the beacons you want to monitor (the proximity UUID is required; the major and minor values are optional). You must also supply a string that uniquely identifies the region so that you can refer to it in your code.

Note

A region’s identifier is unrelated to the identifying information that a beacon advertises.

To register a beacon region, call the startMonitoringForRegion: mothod of your RECOBeaconManager object.

- (void)registerBeaconRegionWithUUID:(NSUUID *)proximityUUID
                       andIdentifier:(NSString*)identifier {

    // Create the beacon region to be monitored.
    RECOBeaconRegion *recoRegion = [[RECOBeaconRegion alloc]
        initWithProximityUUID:proximityUUID
                   identifier:identifier];

    // Register the beacon region with the RECOBeaconManager
    [self.recoManager startMonitoringForRegion:recoRegion];
}

Monitoring of a beacon region begins immediately after registration for authorized apps. When a user’s device detects a beacon that is advertising the identifying information defined by the registered beacon region (proximity UUID, major value, and minor value), the system generates an appropriate region event for your app.

Note

It’s not unusual to configure a beacon region using only a UUID value. Doing so yields a region that is activated when the device comes within range of any beacon with the specified UUID. Upon entering the beacon region, you would then begin ranging for beacons to obtain detailed information about the specific beacons that are nearby. If this value is nil, UUID will be set to the one provided by Perples Inc. by default.

Handling Boundary-Crossing Events for a Beacon Region¶

When a user enters the registered beacon region, the RECOBeaconManager calls the recoManager:didEnterRegion: of its delegate object. Similarly, when a user is no longer in range of any beacons in the registered beacon region, the RECOBeaconManager calls the recoManager:didExitRegion: of its delegate object.

You can specify which boundary-crossing events should notify your app by setNotifyOnEntry and setNotifyOnExit method of RECOBeaconRegion class.

Note

If the device is already inside of the region When the moment of calling startMonitoringForRegion: method, RECOBeaconManager does not call the recoManager:didEnterRegion: of its delegate object. There must be status change to call an appropriate delegate method.

Ranging RECOBeaconRegion¶

Determining the Proximity of a Beacon Using Ranging¶

While a user’s device is inside a registered beacon region, apps can use the startRangingBeaconsInRegion: method of the RECOBeaconManager class to determine the relative proximity of one or more beacons in the region and to be notified when that distance changes.

Note

Always call the isRangingAvailable class method of the RECOBeaconManager class before attempting to range beacons in a beacon region.

Knowing the relative distance to a beacon can be useful for many apps. For example, imagine a museum that places a beacon at each exhibit. A museum-specific app could use a particular exhibit’s proximity as a cue to provide information about that exhibit rather than another.

The RECOBeaconManager calls the locationManager:didRangeBeacons:inRegion: of its delegate object whenever beacons in the specified beacon region come within range, go out of range, or their proximity changes. This delegate method provides an array of RECOBeacon objects that represent the beacons currently in range. The array of beacons is ordered by approximate distance from the device, with the closest beacon at the beginning of the array. You can use the information in these objects to determine the proximity of the user to each beacon. The value in the proximity property of the RECOBeacon object gives a general sense of the relative distance to a beacon.

Note

Beacon ranging depends on detecting the strength of Bluetooth low-energy radio signals, and the accuracy of those signals is attenuated (or lessened) by walls, doors, and other physical objects. The signals are also affected by water, which means the human body itself will affect the signals. It is important to be aware of these factors when planning your iBeacon deployment because they will impact the proximity value reported by each beacon. If needed, use the accuracy and rssi values reported by each RECOBeacon object to adjust the placement of your beacons during deployment.

// delegate method of RECOBeaconManager class
- (void)recoManager:(RECOBeaconManager*)manager
        didRangeBeacons:(NSArray *)beacons
               inRegion:(RECOBeaconRegion*)region {

    if ([beacons count] > 0) {
        RECOBeacon *nearestExhibit = [beacons firstObject];

        // Present the exhibit-specific UI only when
        // the user is relatively close to the exhibit.
        if (RECOProximityNear == nearestExhibit.proximity) {
            [self presentExhibitInfoWithMajorValue:nearestExhibit.major.integerValue];
    } else {
        [self dismissExhibitInfo];
    }
}

Note

If multiple beacon devices are advertising the same combination of proximity UUID, major values, and minor values, they may be reported by the recoManager:didRangeBeacons:inRegion: method as having different proximities and accuracies. It’s recommended that each beacon device be uniquely identified.

How to use RECO SDK¶

The following describes how to use RECO SDK.

  1. Import SDK
#import <Reco/Reco.h>
  1. Implement delegate in accordance with protocol
- (void) recoManager:(RECOBeaconManager *)manager didEnterRegion:(RECOBeaconRegion *)region {
    NSLog(@"Background: didEnterRegion %@", region.identifier);
    // Notification when a user enters the region
}

- (void) recoManager:(RECOBeaconManager *)manager didExitRegion:(RECOBeaconRegion *)region {
    NSLog(@"Background: didExitRegion %@", region.identifier);
    // Notification when a user exits the region
}

- (void)recoManager:(RECOBeaconManager *)manager didRangeBeacons:(NSArray *)beacons
                                                        inRegion:(RECOBeaconRegion *)region {
    NSLog(@"didRangeBeaconsInRegion: %@", ranged %lu beacons",
            region.identifier, (unsigned long)[beacons count]);
    // The information of beacons in the region is passed through (NSArray *)beacons
    // in the form of  RECOBeacon objects.
}
// etc.
  1. Make an instance of RECOBeaconManager class.
RECOBeaconManager *recoManager = [[RECOBeaconManager alloc] init];
  1. Define a RECOBeaconRegion
NSUUID *proximityUUID = [[NSUUID alloc] initWithUUIDString:@""];
NSString *identifier = @"myRecoRegion";
RECOBeaconRegion *recoRegion = [[RECOBeaconRegion alloc] initWithProximityUUID:proximityUUID
                                                                    identifier:identifier];

// Notify on both when user enters and exits the region.
[recoRegion setNotifyOnEntry:YES];
[recoRegion setNotifyOnExit:YES];
  1. Get notified with the corresponding delegate method when monitoring and ranging.
// For monitoring
[recoManager startMonitoringForRegion:recoRegion];

// For ranging
[recoManager startRangingBeaconsInRegion:recoRegion];
  1. Stop monitoring and ranging.
[recoManager stopMonitoringForRegion:recoRegion];

[recoManager stopRangingBeaconsInRegion:recoRegion];

please refer to the sample project for the details.

Behaviors of Monitoring¶

There are two events for monitoring as a user enters or exits the region. User’s location is determined by iOS itself, there might be a time lag as following.

  • When monitoring regions and an application is in background, the time lag for an event to occur might be maximum 15 minutes.
  • In most cases, the time lag is much shorter than 15 minutes. When we, Perples Inc, tested our SDK, the region enter event usually occurred within 30 seconds, and the region exit event occurred with in 1 minutes.

Behaviors of Ranging¶

There is only one event for ranging as a user ranges the region.

  • When ranging regions, recoManager:didRangeBeacons:inRegion: method gets called every second.
  • Ranging is not available in the background basically. However, it is possible to run in the background when it comes with monitoring activity in the background. The iOS gives 10 seconds when recoManager:didEnterRegion: or recoManager:didExitRegion: methods get called while the application runs in the background. Thus, you can range beacons 10 seconds in the region that is received from recoManager:didEnterRegion: or recoManager:didExitRegion: methods even in the background.

iOS 8 Location Service¶

In iOS 8 or later, Location Service authorization types have been changed, which affect every application using the CoreLocation framework in conjunction with the iOS 8 SDK. Since RECO SDK is based on CoreLocation framework, the change of the authorization status affects how to use our RECO SDK to request permission to access Location Services. If you want to build your application using iOS 8 SDK and our RECO SDK, you must read this and use methods to reqeust authorization.

Location Service Authorization Model for iOS 8¶

In iOS 7 or earlier, there are only two types to access location service - allow or deny. In addition, background CoreLocation activities such as region monitoring required Background App Refresh to be enalbed.

In iOS 8 or later, you can control access to location service more fine-grained level, and Background App Refresh is not required any more.

  • Always: The application is authorized to start location services at any time (both of in the foreground and background). This authorization allows you to use all location services, including those for monitoring regions and significant location changes. In addition, services that allow the application to be launched in the background continue to do so.
  • When In Use (While Using the App): The application is authorized to start most location services while running in the foreground. This authorization does not allow you to use APIs that could launch your app in response to an event, such as region monitoring and the significant location change services.
  • Denied (Never): The user explicitly denied the use of location services for the application or location services are currently disabled in Settings.

Authorization Status for Monitoring and Ranging RECOBeaconRegion with iOS 8¶

  • Always: The application can monitor regions and range beacons in regions while the application is running in the foreground or background.

    • To request Always permission, you must call requestAlwaysAuthorization method. This method runs asynchronously and prompts the user to grant permission to the application to use location services. The user prompt contains the text from the NSLocationAlwaysUsageDescription key in your app’s Info.plist file, and the presence of that key is required when calling this method. After the status is determined, the location manager delivers the results to the delegate’s recoManager:didChangeAuthorizationStatus: method.

    • Info.plist file (just before </dict> line):

      <key>NSLocationAlwaysUsageDescription</key>
<string>description for this permission</string>

  • When In Use (While Using the App): The application can range beacons in regions in the foreground, but the application CANNOT monitor regions even in the foreground.

    • To request WhenInUse permission, you must call requestWhenInUseAuthorization method. This method runs asynchronously and prompts the user to grant permission to the application to use location services. The user prompt contains the text from the NSLocationWhenInUseUsageDescription key in your app’s Info.plist file, and the presence of that key is required when calling this method. After the status is determined, the location manager delivers the results to the delegate’s recoManager:didChangeAuthorizationStatus: method.

    • Info.plist file (just before </dict> line):

      <key>NSLocationWhenInUseUsageDescription</key>
<string>description for this permission</string>

  • Denied (Never): The application cannot perform monitoring and ranging regions.

How to use RECO SDK¶

The following describes how to use RECO SDK to request authorization.

  1. Import SDK
#import <Reco/Reco.h>
  1. Implement delegate in accordance with protocol
- (void)recoManager:(RECOBeaconManager *)manager
    didChangeAuthorizationStatus:(CLAuthorizationStatus)status {
        NSLog(@"didChangeAuthorizationStatus: status - %d", status);
        // do something when authorization status is changed
    }
  1. Make an instance of RECOBeaconManager class and set delegate.
RECOBeaconManager *recoManager = [[RECOBeaconManager alloc] init];
recoManager.delegate = self;
  1. request authorization
[recoManager requestAlwaysAuthorization];

or

[recoManager requestWhenInUseAuthorization];
